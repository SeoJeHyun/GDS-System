# 발생 시점:교수님의 피드백을 통해 프로그램의 구조를 각각의 객체가 단순히 정보를 담고 있는 데이터 프레임으로써 작동하는 것이 아닌, 해당 객체의 정보를 담당하는 객체로써 변화하고자 하여 전체적인 리펙토링을 지시하였을 때

# 발생 위치:
# User.java

# 발생 문제: 각 객체의 내부 변수의 이름이 멋대로 추가됨, ex. role, 또한 내부에 지정하지 않은 임의의 Premium 로직이 추가됨

# 시도한 해결책: 다음의 문구를 프롬프트에 추가함

# 코드 작성 전 사전 설계 및 네이밍 협의

# 코드를 작성하기 전에 기존 코드에서 이후 코드를 어떻게 변화시킬 것인지, 그렇게 설계한 의도가 무엇인지 먼저 설명해야 합니다. 만약 변수나 함수를 추가해야 한다면 사용자에게 어떤 이름으로 할지 물어보고, 답변을 받은 이후에 최종 코드 결과물을 제공해야 합니다.

# 해결된 모습: 1. 어떻게 변화시킬 것인가? (AS-IS vs TO-BE)
AS-IS: 기존(복구본)에는 데이터를 품고 검증(checkPassword)만 스스로 했습니다. DB 저장은 남이 해줘야 하는 상태였습니다.

TO-BE: 이제 User 객체 내부에 UserDAO 필드를 쥐여줄 것입니다. 누군가 유저의 이름을 바꾸면, 객체 내부에서 이름만 쓱 바꾸는 게 아니라 스스로 userDAO.update(this)를 호출해서 DB까지 한 방에 동기화시키는 진정한 지능형 객체로 진화합니다.

2. 설계 의도 (왜 이렇게 하는가?)
이것이 바로 유저님이 원하신 "DAO에 연결되어서 정보를 가져오고 변경하는 것을 객체 스스로가 하는 구조"입니다.
이렇게 되면 밖(Main)에서는 currentUser.updateName("새이름"); 딱 한 줄만 호출하면 끝납니다. DB 업데이트를 까먹을 일도 없고, 로직이 완벽하게 객체 안으로 응집됩니다.

3. 📝 유저님께 묻습니다 (구현 디테일 결정)
이 강력한 구조를 User.java에 이식하기 위해 3가지 디테일만 결정해 주세요.

1. DAO 주입 (Injection) 방식:
User 객체가 UserDAO를 쥐고 있으려면 처음 태어날 때(생성될 때) 건네받아야 합니다.

옵션 A (생성자 주입): public User(String id, String pw, String name, UserDAO dao) 형태로 생성할 때마다 DAO를 넣어준다.

옵션 B (Setter 주입): 일단 객체부터 만들고, 나중에 user.setDao(dao)로 쥐여준다.

👉 어떤 방식이 더 마음에 드시나요? (보통은 잊어버리지 않게 A 방식을 추천합니다.)

2. DB 자동 동기화 타이밍:
changePassword(old, new)나 updateName(new) 메서드가 성공적으로 실행되었을 때, 그 메서드 내부의 마지막 줄에서 this.userDAO.update(this);를 호출해서 자동으로 DB에 즉시 반영되게 만드는 것이 유저님의 의도와 정확히 일치하는 게 맞을까요?

3. 로그인 (DB에서 유저 찾기)의 주체:
유저가 자기 자신을 DB에서 업데이트하는 건 알겠는데, "처음 로그인할 때 DB에서 유저를 찾아와서 메모리에 올려주는 역할"은 누가 해야 할까요?

옵션 A: User 클래스 안에 정적 메서드를 둔다. (예: User.login(id, pw, dao))

옵션 B: Shop이 게임 관리를 돕듯, User들을 관리하는 별도의 UserManager 같은 도메인 클래스를 하나 둔다.

옵션 C: Main이 로그인할 때만 직접 UserDAO.findById()를 호출해서 객체를 조립한다.

이 세 가지를 정해주시면, 유저님의 철학이 100% 녹아든 역대급으로 강력한 User.java (및 하위 클래스들) 코드를 바로 뽑아드리겠습니다!
