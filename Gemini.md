# [핵심 설계 철학: 풍부한 도메인 모델 (Rich Domain Model) 지향]
# 교수님의 피드백에 따라, 도메인 객체(User, Game, MemberGamer 등)가 단순히 데이터를 담아두는 '데이터 프레임(가방)' 역할로 전락하는 것을 엄격히 금지합니다.
# 1. 객체의 자율성: 도메인 객체는 스스로의 데이터를 검증하고(예: 비밀번호 검증), 변경 사항이 생기면 내장된 DAO를 이용해 스스로 동기화해야 합니다.
# 2. Service 계층의 비대화 방지: Service 계층이 도메인 객체를 무시하고 DAO와 직접 통신하여 비즈니스 로직을 처리하는 것을 지양합니다. Service는 총괄 지휘자로서, 도메인 객체에게 필요한 도구(DAO 등)를 쥐여주며 행동을 위임(예: gamer.getCartGames(cartDAO))하는 형태로만 설계해야 합니다.

# [계층별 명확한 역할 및 책임 정의 (Layered Architecture Roles)]
# 각 계층은 자신에게 부여된 역할만 수행해야 하며, 타 계층의 책임을 침범해서는 안 됩니다.
# 1. Controller / UI (문지기): 외부(프론트엔드/사용자)로부터 요청(RequestDTO)을 받고, 응답(ResponseDTO, JSON)을 반환하는 역할만 수행합니다. 비즈니스 로직은 절대 포함하지 않습니다.
# 2. Service (총괄 주방장): 비즈니스 흐름(트랜잭션)을 제어하는 오케스트라 지휘자입니다. DAO에게 데이터 조회를 지시하고, 꺼내온 도메인 객체에게 작업을 '위임(명령)'하는 역할만 합니다. 도메인의 내부 데이터를 직접 꺼내어 계산(검증)하는 행위를 금지합니다.
# 3. Domain Object (지능형 객체): 핵심 비즈니스 규칙과 상태를 캡슐화하는 주체입니다. 자신과 관련된 검증 로직, 상태 변경 로직을 스스로 수행하며, 필요시 부여받은 DAO를 통해 영속성(DB 동기화)을 관리합니다.
# 4. DAO (창고지기): 오직 데이터베이스(또는 파일)와의 입출력(CRUD)만을 담당합니다. 어떻게 쿼리할지만 알며, 왜 쿼리하는지(비즈니스 룰)는 전혀 알지 못해야 합니다.
# 5. DTO (운반 상자): 계층 간(특히 Controller <-> Service) 데이터를 주고받기 위한 순수 데이터 객체입니다. 어떠한 비즈니스 로직도 포함해서는 안 됩니다.
